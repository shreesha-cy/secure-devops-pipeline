2. Executive Summary
Modern software teams have adopted CI/CD pipelines to ship code faster — but security has historically been an afterthought bolted on at the end, like installing an alarm system after a burglary. This structural disconnect means that vulnerabilities born in the developer's editor only surface after deployment, when the cost to fix them is estimated to be 30× higher than at the point of introduction.
SDOP-2025 solves this by treating every pipeline stage as a mandatory security gate. The analogy is an airport security model: passengers (code commits) must pass through multiple independent checkpoints (SAST, SCA, secret detection, IaC scanning, container hardening, policy enforcement, DAST) before boarding the flight (deployment). Any checkpoint failure grounds the flight immediately, with a clear report on what triggered the hold.
The project is built on a Java Spring Boot application — deliberately chosen because it mirrors real enterprise environments — and orchestrated through GitHub Actions / GitLab CI, Docker, and Kubernetes (Helm). Findings are centralised in DefectDojo and visualised via Grafana dashboards, giving both developers and auditors a single pane of glass for security posture.
Delivered by a four-member team in four weeks, SDOP-2025 demonstrates that shift-left security is not just a theoretical ideal but a concrete, reproducible engineering practice achievable at academic scale.
3. Problem Statement & Motivation
3.1 Core Problem
Security in most CI/CD workflows exists as a final audit rather than a pipeline-native enforcement mechanism. Tools like SAST scanners and DAST proxies exist in isolation — they are not wired to block the pipeline, their output is not centralised, and feedback to developers arrives too late for low-cost remediation. The result is a systemic vulnerability introduction pattern: hardcoded credentials, CVE-laden dependencies, misconfigured Kubernetes manifests, and non-compliant container images all reach production before any security team reviews them.
3.2 Research-Identified Gaps
Gap
Explanation
Tool Isolation
SAST, DAST, and SCA tools operate independently — no unified enforcement pipeline exists in most academic or small-team environments.
Late Vulnerability Detection
Findings surface post-deployment where remediation cost is 30× higher than at the code-commit stage.
Missing Policy Gates
Few systems enforce automated pipeline blocking based on CVSS severity thresholds. Most generate reports that humans must manually act on.
Incomplete Coverage
Published research focuses on SAST/DAST only; IaC security, secret detection, container hardening, and policy-as-code are routinely absent.
Poor Feedback Loop
Developers receive no real-time, actionable feedback within their normal workflow, forcing high context-switch cost to act on findings.
Cost & Complexity
Enterprise DevSecOps platforms (Snyk Enterprise, Veracode) are prohibitively expensive and over-engineered for academic or startup contexts.
3.3 Why This Problem Matters for Our Context
As CSE Cyber Security students, the pipeline we build represents exactly the kind of system we will be asked to design, audit, or extend in industry. A working SDOP-2025 demonstrates not just academic competency but practical engineering fluency across the DevSecOps stack — making it directly relevant to placement interviews, portfolio projects, and real-world assignments at companies like HPE, Atlassian, and Adobe.

6. System Architecture & Pipeline Design
6.1 Architecture Philosophy: Sequential Security Gate Model
The pipeline is designed as a sequential gate model — analogous to an assembly line quality check, where each station must pass before the product moves forward. Unlike parallel scanning models (faster but weaker enforcement), every stage in SDOP-2025 is a blocking checkpoint. A failure at Stage 2 (SAST) terminates the pipeline immediately, preventing wasted compute and, more importantly, preventing a false sense of security from partial completion.

8. Functional Requirements
8.1 Pipeline Trigger Requirements
FR-01: The pipeline MUST trigger automatically on every git push to any branch
FR-02: The pipeline MUST trigger automatically on every Pull Request (PR) / Merge Request (MR) creation or update
FR-03: No stage may be skipped, including for hotfix branches — all 8 stages are mandatory
FR-04: The pipeline MUST complete all pre-build stages (1–4) before a Docker image is built
8.2 Security Gate Requirements
FR-05: Stage 1 — Any secret pattern match (entropy-based or regex) MUST fail the pipeline
FR-06: Stage 2 — SonarQube Quality Gate = FAILED status MUST fail the pipeline
FR-07: Stage 3 — Any CVE with CVSS >= 7.0 in direct or transitive dependency MUST fail the pipeline
FR-08: Stage 4 — Any Checkov HIGH severity IaC finding or Hadolint DL3/DL4 rule violation MUST fail the pipeline
FR-09: Stage 5 — Docker image CIS benchmark compliance < 80% MUST fail the pipeline
FR-10: Stage 6 — Any OPA Rego policy DENY evaluation MUST fail the pipeline; at minimum the following policies are enforced:
Container must run as non-root user (UID != 0)
CPU and memory resource limits must be set on all containers
Only images from approved registries (Docker Hub official, GHCR) are permitted
FR-11: Stage 7 — Any OWASP ZAP alert rated HIGH or CRITICAL MUST fail the pipeline
8.3 Reporting Requirements
FR-12: All scan findings from all stages MUST be imported into DefectDojo automatically via API
FR-13: Grafana dashboard MUST display: total pipeline runs, pass/fail rate, critical finding count per run, mean time to detect (MTTD), and per-stage failure frequency
FR-14: Slack notification MUST fire within 60 seconds of any pipeline stage failure, containing: failing stage name, severity, finding summary, and link to DefectDojo finding
FR-15: Each pipeline run MUST produce a unified HTML/JSON security report artifact downloadable from the CI/CD platform

OPA Rego policies must be modular. Adding a new enforcement rule should require only adding a .rego file — no pipeline YAML changes.
Observability
Dashboard refresh < 30 sec
Grafana panels must reflect latest pipeline run data within 30 seconds of pipeline completion.
Scalability
Supports >= 3 parallel PR runs
Pipeline must not serialize on shared resources. Docker containers per stage ensure isolation and concurrency.

15. Glossary
Term
Definition
CI/CD
Continuous Integration / Continuous Deployment — automated pipeline for building, testing, and deploying software on each code change
SAST
Static Application Security Testing — analysis of source code or bytecode without execution to detect vulnerabilities
DAST
Dynamic Application Security Testing — testing a running application by sending malicious inputs to detect runtime vulnerabilities
SCA
Software Composition Analysis — scanning third-party libraries and dependencies for known CVEs
IaC
Infrastructure-as-Code — managing and provisioning infrastructure via machine-readable configuration files (e.g., Dockerfiles, K8s manifests, Helm charts)
CVSS
Common Vulnerability Scoring System — a 0–10 score for vulnerability severity. >= 7.0 = HIGH; >= 9.0 = CRITICAL
OPA
Open Policy Agent — an open-source policy engine that evaluates Rego-language policies against JSON/YAML inputs
Shift-Left
The practice of moving security activities earlier (left) in the SDLC — from deployment-time audits to development-time enforcement
CIS Benchmark
Center for Internet Security hardening guidelines — industry-standard configuration baselines for Docker, Kubernetes, OS, etc.
DefectDojo
An open-source vulnerability management platform that aggregates findings from multiple security tools and tracks remediation
Helm
Kubernetes package manager — uses charts (parameterised YAML templates) to define, install, and upgrade Kubernetes applications
Rego
The policy language used by OPA — a declarative logic language for expressing access control and compliance policies
